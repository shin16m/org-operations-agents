# 要件定義書 — 記入チェックリスト

| 版 | 1.0 |
| 日付 | 2026-06-10 |
| SSOT 親 | [`requirements-document-format.md`](requirements-document-format.md) |
| 用途 | 記入完了確認 · レビュー前のセルフチェック |

## 使い方

1. 要件定義書の初稿ができたら、下表を上から順に確認する
2. **必須**がすべて ○ になるまで記入を続ける
3. **推奨**は可能な限り ○ にする（図表なしでも必須 6 節は完遂可能）

---

## 必須チェック（6 節）

| # | 節 | 確認項目 | ○ |
|---|-----|----------|---|
| 1 | ツール名 | 正式名称が記載されている | ☐ |
| 1b | ツール名 | 識別子 / slug（パス）が記載されている | ☐ |
| 2 | システム概要 | 目的（1–3 文）がある | ☐ |
| 2b | システム概要 | 利用者が明示されている | ☐ |
| 2c | システム概要 | スコープ（含む / 含まない）がある | ☐ |
| 3 | 機能一覧 | 機能が 1 行以上の表で列挙されている | ☐ |
| 3b | 機能一覧 | 各機能にトリガー・入力・出力がある | ☐ |
| 4 | I/O 一覧 | 接続先（API・ファイル・サーバー）が 1 件以上ある | ☐ |
| 4b | I/O 一覧 | 認証・秘密情報は**名前のみ**（値は書いていない） | ☐ |
| 4c | I/O 一覧 | エンドポイントまたはファイルパスが具体である | ☐ |
| 5 | 操作手順 | 前提条件・手順・期待結果がある | ☐ |
| 5b | 操作手順 | 安全注意（削除・本番影響等）がある | ☐ |
| 6 | トラブルシューティング | 症状・想定原因・対処が表または箇条書きである | ☐ |
| 6b | トラブルシューティング | エスカレーション先・条件がある | ☐ |

---

## 推奨チェック（図表・品質）

| # | 種別 | 確認項目 | ○ |
|---|------|----------|---|
| D1 | 図 | システム構成図またはアーキテクチャ図がある | ☐ |
| D2 | 図 | I/O データフロー図がある | ☐ |
| D3 | 図 | 操作フロー図がある（手順が 3 段階以上の場合） | ☐ |
| D4 | 表 | 機能一覧が表形式である | ☐ |
| D5 | 表 | I/O マトリクス表がある | ☐ |
| D6 | 表 | トラブルシューティングが症状→対処表である | ☐ |
| Q1 | 品質 | 専門用語に初出定義または用語集がある | ☐ |
| Q2 | 品質 | 付録に変更履歴または参照リンクがある | ☐ |

---

## 図表ガイド（章別・推奨一覧）

| 章 | 推奨図表 | 必須度 | 代替（図なし時） |
|----|----------|--------|------------------|
| 1. ツール名 | 名称表 | 必須 | テンプレート表のみで可 |
| 2. システム概要 | システム構成図 | 推奨 | 箇条書きでコンポーネント列挙 |
| 3. 機能一覧 | 機能一覧表 | 必須 | — |
| 4. I/O 一覧 | I/O マトリクス · データフロー図 | 表必須 · 図推奨 | 表のみで可 |
| 5. 操作手順 | 操作フロー図 · 実施前チェックリスト | 推奨 | 番号付き手順のみで可 |
| 6. トラブルシューティング | 症状→対処表 · 決定木 | 表必須 · 木推奨 | 表のみで可 |

### 図表フォーマット推奨

| 種類 | 推奨ツール | 備考 |
|------|------------|------|
| 構成図 · フロー | Mermaid（md 内） | Git で diff 可能 |
| 複雑な UI | スクリーンショット | `docs/design/examples/` 配下 |
| 大量 I/O | Markdown 表 | 列: 種別 · 名称 · パス · 認証 |
