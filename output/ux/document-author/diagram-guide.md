# 図表ガイド — document-author 向け

読み手の認知負荷を下げるため、**文字だけで足りないときだけ**図表を使う。

---

## 使い分けの原則

| 手段 | いつ使う | 避ける場面 |
|------|----------|------------|
| **箇条書き** | 3〜7 項目の列挙 | 10 項目超（表へ） |
| **表** | 比較・属性一覧・対応表 | 長文説明の代替 |
| **Mermaid flowchart** | 手順・配賦順・分岐が 3 ノード以上 | 単純な A→B→C（矢印 1 行で足りる） |
| **Mermaid sequence** | 役割間のやりとり（dispatch 等） | 静的な一覧 |
| **ASCII / コードブロック** | CLI・パス・コマンド | ビジュアル説明 |
| **Cursor Canvas** | 大量データ · 多軸比較 · 探索 UI | 短い企画書・1 ページ説明 |

## Mermaid 制約（org-ops 標準）

- ノード ID にスペースを使わない（`camelCase` / `snake_case`）
- ラベルに `()` `[]` を含むときは引用符で囲む
- 1 図あたり **15 ノード以下**を目安（超えるなら章を分ける）
- 色指定・`style` / `classDef` は使わない（レンダラー互換のため）

## テンプレ別の推奨図

### 汎用説明文書

- プロセスが 4 段以上 → `flowchart LR` 1 枚
- 比較 2 案以上 → 表

### 企画書

- 配賦順・マイルストーン → `flowchart TB` with `subgraph`
- 子タスクは **表**（図にしない）

### システムカタログ

- 全体アーキ → `flowchart LR`（orchestrator → PM → worker → output）
- slug 一覧は **表**（図にしない）

## 配置ルール

1. **概要の直後**に「全体像」図を 1 枚まで
2. **詳細の各章**に必要なら図 1 枚
3. 図の直後に **1〜2 文のキャプション**（図だけで終わらない）

## Canvas 活用の可否

| 推奨 Canvas | Markdown + Mermaid で十分 |
|-------------|---------------------------|
| 請求・メトリクス調査 | 企画書・Epic サマリ |
| セキュリティ監査の証跡一覧 | システムカタログ（表中心） |
| 多チーム成果物の横断比較 | 単発レポート（5 ページ未満） |

**判断:** 読者が**スクロール 1 画面で把握**できるなら MD。探索・ソート・フィルタが必要なら Canvas を document-author の成果物オプションとして提案する。

## 例: 最小 flowchart

```mermaid
flowchart LR
  input[インプット] --> author[document-author]
  author --> doc[説明文書MD]
  doc --> reader[読者]
```
